Market Research Report
Executive summary
Your current product should be treated as a prototype with partial integrations, not a production
marketing and automation platform. The core technical gap is not “missing buttons”; it is the lack of a
reliable control plane for OAuth, token lifecycle, browser-session restoration, publish orchestration, asset
generation, and observability. The highest-leverage technical move is to standardize on a backend-owned
integrations platform: every official app connection starts from your frontend, completes on your backend
with Authorization Code flow, stores encrypted tokens server-side, refreshes automatically, and exposes
only internal tool endpoints to the AI agent. This is how Google web-server OAuth, Slack OAuth v2, LinkedIn
3-legged OAuth, WordPress.com OAuth2, and similar providers are designed to be used. Google explicitly
documents server-side OAuth flows, token storage best practices, and refresh handling; Slack requires
redirect URIs to be configured consistently; LinkedIn and Notion also center their app integrations on user-
authorized OAuth flows. 
The browser agent should not be your primary publisher for providers that already support official APIs. It
should be a fallback execution layer for flows that are unsupported, blocked by permission review, or
operationally brittle. Playwright’s own guidance makes this separation natural: authenticated browser state
can be stored and reused, but those files are sensitive and must not be committed; locators should rely on
resilient selectors; traces, screenshots, and videos should be captured on failure to debug automation
problems. In practice, that means LinkedIn, some WordPress flows, and account-recovery/verification flows
can use the browser agent, while Gmail, Calendar , Notion, Slack, Meta Pages/Instagram, X, SendGrid, and
image providers should default to official APIs wherever possible. 
Your marketing engine also needs a hard split between generation, review, publishing, and 
measurement. The engine should never fabricate success. If an image provider fails, the campaign must
show image_generation_failed and stop or degrade cleanly; if a social publish returns 429 or a
permissions error , the attempt should be recorded as failed with retriable metadata; if browser automation
lands on a login page, that is a stale-session event, not a successful publish. This is especially important
because Google, Slack, X, SendGrid, Replicate, Stability, and other providers all publish rate-limit or quota
behavior that should drive real retry logic and user-facing status. 
For the market research track, the best initial vertical is local home services—specifically HVAC, plumbing,
electrical, and roofing contractors—because the segment is large, fragmented, locally marketed, lead-
response dependent, and operationally close to the workflows your product can automate: inbound lead
capture, quoting, appointment scheduling, review requests, local social posting, ad landing-page testing,
email nurture, and follow-up. Official Census SUSB data shows that U.S. firms in plumbing/HVAC, electrical
contracting, and roofing together total roughly 210,659 firms and 215,975 establishments, with about 
189,492 firms in businesses under 20 employees, which is exactly the kind of fragmentation where a
focused, verticalized AI growth OS can win. BLS also shows meaningful ongoing demand across these

Product diagnosis and target operating model
Your current symptoms are consistent with a missing production backbone: LinkedIn partly works because
one connector is wired to a valid OAuth or browser path, while other connectors are blocked because their
app configs, scopes, redirect URIs, and token-exchange code are incomplete. The browser agent stopping
on login pages indicates that session bootstrap, expired-state detection, MFA checkpointing, and human-in-
the-loop recovery are not implemented as first-class states. The campaign engine producing fake success or
non-sendable email implies that generation and publishing are not being validated against provider
responses and deliverability prerequisites such as sender verification. SendGrid, for example, requires
authenticated or verified senders; Google and Meta enforce scope- and review-based access; LinkedIn
access depends on approved products and scopes.

Technical production hardening track
Integration architecture and OAuth control plane
The production rule is simple: frontend starts connection, backend owns OAuth, backend stores
encrypted tokens, AI uses backend tools. For official OAuth providers, use Authorization Code flow for
confidential clients on the server . For X, prefer OAuth 2.0 Authorization Code with PKCE for user context. For
Notion, use a public connection acting on behalf of the signed-in user . For WordPress.com, use OAuth2 for
WordPress.com sites and Application Passwords for self-hosted WordPress unless you explicitly add a site-
side OAuth plugin.

Production checklist for the OAuth control plane:
generate and persist state per connect attempt
verify state on callback
store requested scopes and returned scopes
encrypt access_token and refresh_token
calculate and persist expires_at
refresh access tokens server-side before expiry
revoke and delete tokens on disconnect
separate provider credentials by environment
never expose tokens, client secrets, or refresh tokens to the frontend
capture all integration events in an audit table
Google’s published best practices are explicit here: keep client credentials in secure storage, keep user
tokens secure, never transmit them in plaintext, and revoke/delete when no longer needed.
